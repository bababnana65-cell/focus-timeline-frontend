# 客户端与服务端职责说明

## 1. 目标

当前原型阶段允许时间线数据存在本地，但正式产品不能让每台手机各自维护一套“标准时间线”。服务端应成为标准数据中心，客户端负责读取、缓存与交互展示。

## 2. 当前阶段与正式阶段

### 当前阶段

- Flutter 客户端内置 mock 数据
- 用户可在本地创建时间线草稿
- 分享可携带快照数据
- 本地保存关注、历史、草稿与缓存

### 正式阶段

- 服务端保存标准专题、标准节点和专题关联关系
- 客户端通过 API 拉取标准数据
- 分享链接只传 `topic_id`、`version_id` 等标识
- 所有用户读取同一份标准时间线，再叠加自己的关注、历史和个性化设置

## 3. 服务端负责什么

服务端负责“标准答案”，主要包括：

- 标准专题 `Topic`
- 标准事件节点 `EventNode`
- 节点与专题的多对多关联 `TopicEventLink`
- 用户账号、关注关系、分享记录
- 来源保存、审核状态、可信度、版本控制
- 去重、合并、冲突处理、更新发布

服务端的核心职责不是展示，而是：

- 确定一条事实节点是否成立
- 确定它属于哪些专题
- 确定它的时间精度、来源、可信度和当前版本
- 让不同用户看到一致的数据结果

## 4. 客户端负责什么

客户端负责“读取与使用”，主要包括：

- 登录、搜索、关注、分享、浏览
- 本地缓存时间线，提升启动速度
- 离线展示最近浏览过的内容
- 保存用户关注列表、浏览历史、排序和草稿
- 将用户创建的专题草稿提交给服务端

客户端可以临时保存数据，但不负责定义“标准时间线”。

## 5. 为什么 Topic / EventNode / TopicEventLink 要在服务端成立

产品不应假设“一个事件只能有一条时间线”。同一条事实节点可能同时属于多个专题。

示例：

- 专题 A：美伊战争总体进展
- 专题 B：伊朗导弹发射情况
- 专题 C：霍尔木兹海峡航运情况

事实节点：

- 伊朗发射导弹
- 海峡航运预警升级

正确做法不是复制三份节点，而是：

- 服务端只保存一份 `EventNode`
- 再通过 `TopicEventLink` 关联到 A、B、C 等专题

这样可以保证：

- 修改一次，所有专题同步
- 不会因重复复制造成描述冲突
- 后续做搜索、推荐、审核和版本管理更稳定

## 6. 推荐的最小服务端模型

### Topic

- `id`
- `name`
- `tagline`
- `definition_overview`
- `core_keywords`
- `related_keywords`
- `excluded_keywords`
- `start_at`
- `status`
- `version`

### EventNode

- `id`
- `title`
- `summary`
- `detail`
- `event_time`
- `time_precision`
- `importance`
- `source_name`
- `source_url`
- `source_type`
- `confidence_score`
- `review_status`
- `version`

### TopicEventLink

- `id`
- `topic_id`
- `event_node_id`
- `relevance_score`
- `link_reason`
- `is_primary`

### UserFollow

- `id`
- `user_id`
- `topic_id`
- `is_pinned`
- `created_at`

### TopicViewHistory

- `id`
- `user_id`
- `topic_id`
- `viewed_at`

## 7. 分享与同步如何工作

正式版中，分享不应传整份节点正文，而应优先传标识：

- `topic_id`
- `version_id`
- 可选 `entry_id`

用户 A 分享后：

1. 服务端生成可访问链接
2. 用户 B 打开链接
3. 客户端从服务端拉取该专题对应版本
4. 用户 B 再决定是否关注

这样可以同时支持两种模式：

- 查看分享当时的固定版本
- 切换到该专题的最新版本

## 8. 数据准确性由谁保证

准确性和一致性主要由服务端保证，原则包括：

- 保留原始来源与来源链接
- 重要节点尽量双来源确认
- 审核状态分层，例如待审核、已确认、有争议、已驳回
- 同一事实节点不重复建多份
- 内容修正走版本机制，不直接覆盖历史

客户端只负责展示这些状态，不应本地发明新的“标准事实”。

## 9. 当前原型应如何配合未来架构

在没有后端的阶段，客户端可以先继续：

- 用本地 mock 数据做演示
- 用本地草稿模拟专题创建流程
- 用本地缓存模拟历史和关注

但需要记住：

- 当前本地结构只是过渡方案
- 标准专题、标准节点、关联关系最终都应迁到服务端
- 客户端未来应改成“拉取 + 缓存 + 提交操作”，而不是“自己维护标准库”

## 10. 推荐路线

建议按这个顺序推进：

1. 继续完成客户端原型和交互
2. 定清服务端数据模型与 API
3. 接入手机号短信登录
4. 接入标准专题、标准节点与多专题关联
5. 最后再做自动采集、去重与审核后台
