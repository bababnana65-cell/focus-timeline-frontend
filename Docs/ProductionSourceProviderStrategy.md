# Timeliness 正式版信息源与 SourceProvider 策略

更新时间：2026-05-11

## 目标

Timeliness 的核心不是百科查询，而是帮助用户持续追踪近期仍可能更新的事件时间线。正式版信息源设计应优先服务三个目标：

1. 快速判断用户输入能否形成可持续追踪的时间线。
2. 为时间轴节点提供可验证的新闻、官方公告或权威来源。
3. 在不拖慢创建页体验的前提下，后台持续发现新节点，并通过关注列表红点和高亮提醒用户。

## 总体原则

不要依赖单一新闻 API。正式版应采用 SourceProvider 分层架构：

```text
中文新闻聚合源
+ 搜索源
+ 官方源/RSS
+ 垂直行业源
+ 社媒热度源
+ 知识背景源
-> 统一归一化
-> 去重、相关性过滤、可信度评分
-> AI 归纳为候选方向或时间轴节点
```

AI 不应直接编造最新事实。信息源负责提供事实线索和链接，AI 负责归纳、去重、判断是否属于用户确认的时间线范围，并生成标题、摘要和节点解释。

## SourceProvider 分层

### 1. 中文新闻聚合源

用途：

- 发现中国用户关心的近期新闻。
- 支撑候选方向里的“近期仍有更新”判断。
- 为新节点提供新闻依据。

候选：

- TianAPI / 天行数据
- 聚合数据
- 极速数据
- 六派数据
- 其他具备中文新闻授权的商业 API

注意：

- 很多国内新闻 API 对“终端展示”和“商用分发”有授权限制，正式版必须确认协议。
- 可以用于内部分析、机器学习或事实线索的接口，不等于可以直接把全文展示给用户。
- 正式版应优先展示摘要、来源名、发布时间和原文链接，避免复制大段正文。

### 2. 搜索源

用途：

- 处理用户输入模糊、长尾、非标准关键词的情况。
- 补充新闻 API 覆盖不到的事件。
- 在候选阶段快速理解近期发生了什么。

候选：

- Tavily
- SerpApi Baidu
- Bing Web Search
- 其他合规搜索 API

注意：

- 搜索结果只能作为线索，不能单独作为高置信事实节点。
- 搜索源结果必须经过标题、正文片段、发布时间、来源可信度和专题范围相关性过滤。

### 3. 官方源与 RSS

用途：

- 提供高可信节点。
- 用于政策、外交、监管、安全提醒、上市公告、灾害通报等场景。

候选：

- 中国政府网
- 外交部、商务部、发改委、工信部、证监会等部委网站
- 地方政府官网
- 交易所公告、巨潮资讯
- 企业官网新闻稿、投资者关系页面
- 机构 RSS

注意：

- 官方源更新频率不一定高，但可信度高。
- 官方源更适合节点确认、政策节点、调查处置节点，不适合发现所有热点。

### 4. 垂直行业源

用途：

- 提高特定类别的覆盖和准确性。

建议分领域接入：

- 财经：交易所公告、巨潮资讯、Tushare、AKShare、东方财富类数据源。
- 科技：公司官网、开发者博客、模型发布页、GitHub Release、论文/技术博客。
- 体育：赛事官网、赛程数据、权威体育媒体。
- 灾害天气：气象、应急管理、地震、水利、交通管制等官方源。
- 航运能源：航运、能源、港口、海关、行业协会数据源。

注意：

- 垂直源不必一开始全接，应按用户高频类别逐步补。
- 每个垂直源都要统一输出标准 SourceItem。

### 5. 社媒热度源

用途：

- 判断中国用户关注度。
- 发现突发热点。
- 辅助候选方向排序。

候选：

- 微博热搜
- 百度热搜
- 抖音/头条趋势
- 微信指数、百度指数等趋势数据

注意：

- 社媒热度只能作为热度信号，不能直接作为事实依据。
- 不能因为热度高就写入时间轴节点，必须回到新闻或官方来源确认。

### 6. 知识背景源

用途：

- 补历史背景、前因后果、名词解释。
- 帮助用户理解时间线起点之前的背景。

候选：

- 百科类源
- 官方历史页面
- 机构报告
- 维基类知识源
- 学术、论文、行业报告

注意：

- 知识源不应成为默认主方向。
- 如果近期没有新闻更新，创建页应提示“近期更新较少，适合作为历史时间线阅读”。

## 前台与后台使用方式

### 创建页：快

创建页目标是让用户快速确认“我要看哪条时间线”，不能等待全量来源查询。

建议链路：

```text
用户输入关键词
-> 点击 AI 扩写
-> 并发查询 1 个快速搜索源 + 最近新闻缓存
-> 生成 2-3 个 AI 建议追踪方向
-> 用户选择方向
-> 生成完整草案：标题、简介、类别、AI 理解、起点、关键词
```

创建页不做：

- 不实时查所有来源。
- 不做完整事实校验。
- 不生成完整 20 个节点。
- 不因低证据阻塞创建。

创建页应做：

- 给出候选方向。
- 标注近期活跃状态。
- 提示证据不足或近期更新较少。
- 让用户手动修正类别、起点和关键词。

### 时间轴初始化：较深

创建成功后，后台初始化可以更慢一些，目标是生成首批可信节点。

建议链路：

```text
创建专题
-> 后台读取用户确认的 trackingDirection/topicScope/topicObject
-> 查询新闻源、搜索源、官方源、垂直源
-> 去重与相关性过滤
-> AI 生成 3-7 个初始节点
-> 写入时间轴
-> 若失败则回退 deterministic seed
```

要求：

- 节点必须匹配用户确认的追踪方向。
- 节点应保留 sourceName、sourceUrl、sourcePublishedAt、linkReason。
- 无来源支持的最新事实不能作为高置信节点。

### 后台巡检：持续

用户关注专题后，系统定时刷新信息源，发现新节点。

建议链路：

```text
定时任务
-> 按关注专题读取 topicScope/trackingQuestion
-> 查询近期新闻与官方源
-> 判断是否是新节点
-> 写入时间轴
-> 更新 latestNode/hasRecentUpdate
-> 前端显示红点和关注列表高亮
```

本阶段“推送”的含义是应用内新节点提醒，不等于系统通知。系统推送通知需要后续单独设计。

## 性能策略

正式版不能每次请求都实时查所有来源。必须控制前台延迟。

建议策略：

- Provider 并发查询，不串行等待。
- 创建页只等快速源，目标 1-2 秒返回候选。
- 每个 Provider 单独超时，创建页建议 1.5-3 秒。
- 慢源结果进入后台补充，不阻塞前台。
- 同一关键词或同一 topicScope 缓存 5-30 分钟。
- 热门专题提前预取。
- 已关注专题后台定时刷新。
- 低优先级源失败不影响主流程。

推荐超时分层：

```text
创建页候选：1-2 秒目标，最多 3 秒
完整草案：3-8 秒
初始化节点：30-60 秒后台任务
后台巡检：分钟级或小时级，不阻塞用户
```

## 统一数据结构

所有来源应归一化为统一结构，避免业务逻辑绑定供应商字段。

```json
{
  "title": "新闻或公告标题",
  "url": "https://example.com/article",
  "sourceName": "来源名称",
  "publishedAt": "2026-05-11T10:00:00+08:00",
  "summary": "来源摘要",
  "contentSnippet": "正文片段",
  "sourceType": "news",
  "reliability": "medium",
  "language": "zh",
  "country": "CN",
  "provider": "tavily"
}
```

`sourceType` 建议值：

```text
news
official
rss
search
social
finance
knowledge
company
research
```

`reliability` 建议值：

```text
high    官方公告、交易所公告、权威机构、企业官网
medium  正规媒体、商业新闻 API、可靠行业源
low     搜索片段、社媒热度、未验证转载
```

## 与 AI 的边界

AI 应做：

- 理解用户想看的时间线方向。
- 把来源归纳为 2-3 个候选方向。
- 判断来源是否属于 topicScope。
- 去重相似新闻。
- 生成节点标题、摘要、时间、信号类型和 linkReason。

AI 不应做：

- 编造没有来源支持的最新事实。
- 把百科信息当成近期更新。
- 忽略用户手动选择的类别、起点、关键词和排除词。
- 把静态主题建议成时间线方向。

## 中国用户优先级

面向中国用户时，信息源排序应倾向：

```text
国内中文新闻和搜索
> 官方/机构源
> 垂直行业源
> 社媒热度
> 国际新闻源
> 知识背景源
```

国际新闻仍然重要，尤其是战争、外交、科技公司、金融市场、能源航运等专题。但对于中国用户，候选方向排序应考虑中文语境、国内关注度和中文来源覆盖。

## 测试阶段建议

当前可低成本测试：

```text
Tavily
+ FreeNewsApi
+ APITube 或 NewsData
+ 少量官方 RSS
```

定位：

- Tavily：搜索型上下文。
- FreeNewsApi：免费结构化新闻测试源。
- APITube/NewsData：全球新闻补充。
- 官方 RSS：高可信节点验证。

FreeNewsApi 不适合作为中国正式版主源，但适合测试 SourceProvider 抽象、去重、证据计数、近期活跃判断。

## 正式版最小可行组合

第一阶段建议：

```text
chinese_news_api
tavily_search
official_rss
finance_disclosure
social_trend
knowledge_background
```

不要一次性接入所有供应商。先实现 SourceProvider 抽象和统一数据结构，再按类别逐步补源。

## 风险

- 免费新闻 API 覆盖不稳定，不能作为正式版唯一来源。
- 国内新闻 API 授权条款复杂，必须确认是否允许终端展示。
- 搜索结果可能包含噪声，需要强相关性过滤。
- 社媒热度可能失真，不能直接生成事实节点。
- 多源接入后成本和延迟会上升，必须依赖缓存、并发和后台任务。

## 后续开发顺序

1. 抽象 `SourceProvider` 接口。
2. 把 Tavily 改造成第一个标准 SourceProvider。
3. 接入一个免费新闻 API 作为测试 Provider。
4. 实现多源并发、超时、缓存和归一化。
5. 给候选方向增加 `sourceEvidenceCount`、`latestRelevantSourceAt`、`recentActivityStatus`。
6. 初始化阶段使用多源结果生成 3-7 个来源支持节点。
7. 后台巡检已关注专题，写入新节点并驱动红点。
8. 正式版接入国内中文新闻源、官方源和垂直源。
