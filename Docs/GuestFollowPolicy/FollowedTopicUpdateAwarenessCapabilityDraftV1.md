# 关注专题更新提醒与未读感知能力草案 v1

## 一、目标

当前版本已经具备：

- 关注 / 取消关注
- “我的关注”列表
- 推荐页与时间轴
- 推荐页下拉刷新
- “我的关注”页下拉刷新

下一条新能力线的目标，不是新增消息中心或复杂推送，而是先解决一个最直接的问题：

**用户如何知道自己关注的专题有没有新动态、现在最该先看哪个专题。**

## 二、当前能力基础

后端当前已经具备的相关字段有：

- `lastViewedAt`
- `latestRelevantEventAt`
- `latestRelevantEventSummary`
- `hasRecentUpdate`
- `generatedAt`

因此第一版不需要新建复杂未读体系，可以先基于这些字段完成最小更新感知能力。

## 三、当前版本建议范围

### 1. “我的关注”里明确显示“有更新”

在“我的关注”卡片上强化这些信息：

- 当前专题是否有更新
- 最近更新时间
- 最近动态摘要

目标是让用户一眼看出：

- 哪些专题最近有变化
- 哪个专题更值得现在点进去看

### 2. 已看 / 未看先做最小语义

当前先不做复杂逐条未读，只做专题级更新感知。

规则基于：

- `lastViewedAt`
- `latestRelevantEventAt`
- `hasRecentUpdate`

即：

- 如果最近相关动态时间晚于上次查看时间，则该专题视为“有更新”

### 3. 进入专题后更新已看状态

当前最小规则下，用户进入专题后：

- 后端刷新 `lastViewedAt`
- 前端下次回到“我的关注”时，更新状态应能正确消失或变化

### 4. 推荐页暂不做复杂联动

推荐页可以后续复用“最近更新”的语义，但当前版本先不要求大改推荐页结构。

## 四、明确不做的范围

当前这条能力草案明确不包括：

- 系统推送
- 消息中心
- 复杂未读数模型
- 逐节点已读
- 多设备已读同步优化
- 更复杂的通知优先级系统

这些都不属于当前最小范围。

## 五、后端最小支持建议

### 1. 稳定 `hasRecentUpdate` 语义

继续沿用当前规则：

- 基于 `lastViewedAt`
- 与 `latestRelevantEventAt` 对比
- 由服务端返回 `hasRecentUpdate`

### 2. 保证 `latestRelevantEventSummary` 稳定可用

“我的关注”中每个专题都应尽量返回：

- 最近动态摘要

避免前端只能拿时间，没有内容概览。

### 3. 可选排序增强

后续如有需要，可以考虑为 `GET /topics/followed` 增加轻量排序能力，例如：

- `sort=recentUpdate`

但这不是当前第一版必须项。

## 六、前端最小实现建议

### 1. 强化“我的关注”卡片的更新状态

卡片层应能更明确地区分：

- 有更新
- 无更新

比如：

- 角标
- 高亮文案
- 最近更新摘要
- 最近更新时间

### 2. 列表排序或视觉优先级

即使不新增后端排序参数，前端也可以先在当前结构中更明显地展示“有更新”的专题。

### 3. 不新增新页面

当前不要求新增：

- 通知页
- 消息中心
- 独立未读中心

先把“我的关注”列表本身的信息密度做够。

## 七、联调重点

后续如果进入这条线实现，联调应重点确认：

1. “我的关注”页下拉刷新后，最新动态字段会重新聚合
2. `hasRecentUpdate` 与 `lastViewedAt / latestRelevantEventAt` 语义一致
3. 进入专题后，再回到“我的关注”，状态能正确变化
4. 刷新失败时保留旧数据，不清空页面

## 八、一句话原则

当前版本先把：

**“我的关注里哪些专题有更新”**

这件事做好，而不是一口气上复杂通知系统。

## 九、版本规则

当前文档作为 `v1` 草案使用。

如果后续产品决定扩展这条能力，例如：

- 推送通知
- 消息中心
- 复杂未读数
- 多设备已读同步

则统一新增 `v2` 文档，不直接覆盖当前 `v1`。
