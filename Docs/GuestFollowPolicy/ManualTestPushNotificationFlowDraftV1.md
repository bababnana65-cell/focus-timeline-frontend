# 手动测试推送链路草案 v1

本草案用于定义“关注专题更新手机推送”的**最小可测闭环**。目标不是直接上线完整自动推送，而是先确认：

- 后端能向已注册设备发送一条测试推送
- 前端能收到推送
- 用户点击推送后，能直接进入对应专题

## 1. 目标

当前推送能力分两步走：

1. 已完成：
- 设备 token 注册
- 设备 token disable

2. 当前要补：
- 手动测试推送
- 点击后直达专题

本阶段只解决“能发、能收、能跳转”。

## 2. 当前范围

本阶段只做：

- 面向已登录用户、已注册 push device 的测试推送
- 推送 payload 中携带 `topicId`
- 点击推送后直达对应专题 detail / timeline

本阶段不做：

- 关注专题自动触发推送
- 通知中心
- 推送设置页
- 复杂未读数
- 复杂去重 / 节流

## 3. 最小后端能力

建议后端提供其中一种最小方式：

### 方案 A：开发环境测试接口

新增一个仅开发环境使用的接口，例如：

- `POST /users/push-devices/test-notification`

请求体最小字段：

- `topicId`
- `title`
- `body`

用途：
- 给当前登录用户已启用的设备发送一条测试推送

### 方案 B：内部脚本

如果暂时不想暴露测试接口，也可以先提供一个开发脚本：

- 输入 `userId / topicId / title / body`
- 向该用户的已启用设备发送一条测试推送

当前更推荐方案 A，因为前后端联调更直接。

## 4. 最小 payload 语义

测试推送最小 payload 建议包含：

- `type = topic_update`
- `topicId`
- `title`
- `body`

示例：

```json
{
  "type": "topic_update",
  "topicId": "topic_123",
  "title": "你关注的专题有新动态",
  "body": "美伊战争总体进展出现新的关键节点。"
}
```

## 5. 点击后的最小前端行为

前端收到推送并点击后：

1. 读取 payload 里的 `topicId`
2. 打开对应专题
3. 进入 detail / timeline

当前不要增加：

- 中间跳转页
- 通知中心列表
- 复杂落地页分流

## 6. 联调前提

联调前需满足：

1. 用户已登录
2. `POST /users/push-devices` 已成功
3. 当前设备记录 `enabled = true`
4. 目标专题 `topicId` 可正常打开

## 7. 最小联调检查项

本阶段只验证这 4 个点：

1. 测试推送能发送成功
2. 前端设备能收到推送
3. 点击推送后能直接进入对应专题
4. 进入后专题可正常读取 detail / timeline

## 8. 当前不阻塞的内容

以下内容后置，不阻塞本阶段闭环：

- 自动推送触发规则
- 重大节点筛选
- 置顶专题优先推送
- 推送去重 / 节流细则
- 推送开关设置页

## 9. 一句话结论

本阶段先做**手动测试推送 + 点击直达专题**。先把最小闭环打通，再进入自动触发与策略细化。
