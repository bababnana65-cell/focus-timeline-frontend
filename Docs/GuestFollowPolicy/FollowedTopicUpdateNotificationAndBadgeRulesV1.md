# 关注专题更新提醒与推送规则 v1

## 一、目标

在“我的关注”已有最小更新感知能力的基础上，进一步统一三层提醒语义：

1. 列表内提醒
2. “我的关注”按钮提醒
3. 手机推送

当前版本目标不是一口气做完整通知系统，而是先明确：

- 哪些专题算“有新动态”
- UI 上哪些位置应该提示
- 推送什么时候做、什么时候不做

## 二、三层提醒结构

### 1. 列表内提醒

在“我的关注”列表中，对有新动态的专题显示明显提示。

建议展示：

- 有更新标记
- 最近动态摘要
- 最近更新时间

这是最基础、最优先的一层提醒。

### 2. “我的关注”按钮提醒

底部导航里的“我的关注”按钮，当存在至少一个有更新的关注专题时，应出现提示。

当前建议：

- 第一版先用红点
- 暂不要求复杂数字提醒

也就是：

- 先做“有无提醒”
- 后续再决定是否升级成数量

### 3. 手机推送

当关注专题出现新动态时，App 可以给用户发送推送消息。

但推送属于更复杂的一层能力，建议：

- 先把列表内提醒和按钮提醒做稳
- 推送作为第二阶段能力再接入

## 三、什么算“有新动态”

当前统一以后端语义为准：

- `latestRelevantEventAt > lastViewedAt`
- `hasRecentUpdate = true`

前端应以服务端返回的 `hasRecentUpdate` 为主，不自行发明另一套“有更新”规则。

## 四、“我的关注”按钮什么时候亮

当前建议规则：

- 只要存在任意一个关注专题满足 `hasRecentUpdate = true`
- 底部“我的关注”按钮就显示红点

这条规则简单、稳定，也便于后续扩展。

## 五、列表内专题什么时候提示

列表中只提示当前满足以下条件的专题：

- `hasRecentUpdate = true`

专题卡片层至少能让用户看出：

- 这个专题最近有更新
- 更新大概是什么
- 更新时间是什么时候

## 六、什么时候清掉提示

当前建议最小规则：

- 用户进入该专题并成功打开后
- 后端更新 `lastViewedAt`
- 前端下次回到“我的关注”时
- 如果不再满足 `latestRelevantEventAt > lastViewedAt`，则不再显示“有更新”

## 七、推送建议限制

如果后续接手机推送，建议不要所有动态都推。

至少增加这两个限制：

### 1. 推送范围限制

建议优先只推：

- 置顶专题
- 重大节点
- 或用户显式开启提醒的专题

### 2. 推送去重 / 节流

同一专题短时间内连续有多条动态时：

- 不应连续推送多条
- 应由服务端进行去重或节流

## 八、建议推进顺序

### 第一阶段

先做：

- 列表内提醒
- “我的关注”按钮红点

### 第二阶段

再做：

- 手机推送

这样可以避免一开始把通知系统做得过重。

## 九、当前不做的范围

当前这条规则草案不要求立即实现：

- 通知中心
- 复杂未读数
- 多设备未读同步
- 推送优先级系统
- 推送偏好页

这些都属于后续扩展。

## 十、一句话结论

当前版本最合理的结构是：

- 列表里提示
- “我的关注”按钮提示
- 手机推送后置

先把前两层做稳，再考虑推送。
