# 关注额度与游客合并方案 v1

## 1. 关注额度规则

- `guest`：最多关注 `5` 个专题
- `free`：最多关注 `10` 个专题
- `pro`：最多关注 `50` 个专题

统一原则：

- 浏览、推荐、时间轴、搜索、分享查看：游客可用
- 关注额度：按身份层级限制
- 真正的额度校验必须在后端做，前端只做提示和引导

## 2. 用户层级

建议后端统一成一个字段：

- `account_tier = guest | free | pro`

如果用户未登录：

- 前端本地视为 `guest`
- 后端匿名接口也按 `guest` 规则处理相关提示

如果用户已登录：

- 后端根据账号实际套餐返回 `free` 或 `pro`

## 3. 游客关注的实现方式

第一阶段建议用最小方案：

- 游客关注列表先保存在本地
- 本地最多保存 `5` 个 topicId
- 用户登录后，前端调用“游客关注合并接口”
- 后端负责去重、限额、写入正式账号

不建议第一阶段就做“服务端 guest 账户”。

## 4. 后端需要的接口

建议新增或明确这几个接口：

### `GET /users/capabilities`

返回当前用户能力：

```json
{
  "success": true,
  "data": {
    "accountTier": "free",
    "followLimit": 10,
    "followCount": 3,
    "remainingFollowQuota": 7
  }
}
```

### `POST /topics/{topicId}/follow`

后端统一校验：

- 当前已关注数量
- 是否超过上限
- 是否重复关注

### `POST /users/merge-guest-follows`

请求示例：

```json
{
  "guestTopicIds": ["topic_001", "topic_002", "topic_003"]
}
```

返回示例：

```json
{
  "success": true,
  "data": {
    "mergedTopicIds": ["topic_001", "topic_002"],
    "skippedTopicIds": ["topic_003"],
    "followCount": 10,
    "followLimit": 10,
    "remainingFollowQuota": 0
  }
}
```

## 5. 合并规则

建议第一版固定成：

- 游客关注 + 账号已有关注 = 去重后结果
- 如果合并后超出当前账号上限：
  - 优先保留账号原有关注
  - 再按游客关注加入，直到达到上限
- 如果有专题没能合并成功，要明确提示用户

## 6. 前端交互规则

前端要做的事：

1. 游客模式下允许点“关注”
2. 先把游客关注暂存起来
3. 当用户登录后，调用后端“合并接口”
4. 合并成功后刷新“我的关注”
5. 清掉本地游客暂存

## 7. 页面提示建议

建议前端统一展示剩余额度：

- 游客：`还可关注 2 个专题，登录后可关注 10 个`
- 免费：`还可关注 3 个专题，升级后可关注 50 个`
- 付费：`还可关注 12 个专题`

## 8. 这个方案里谁改什么

后端主改：

- 额度规则
- 合并规则
- 能力接口
- 错误码
- 最终关注写入

前端主改：

- 游客本地关注暂存
- 超限提示
- 登录后触发合并
- 剩余额度展示

## 9. 推荐的落地顺序

1. 后端先加 `accountTier/followLimit` 规则
2. 后端补 `GET /users/capabilities`
3. 后端补 `merge-guest-follows`
4. 前端加游客关注本地暂存
5. 前端加超限提示
6. 前端加登录后自动合并

## 10. 一句话版本

游客 `5` 个、免费 `10` 个、付费 `50` 个，这个方案可行；后端负责“规则和合并”，前端负责“暂存和引导”。
