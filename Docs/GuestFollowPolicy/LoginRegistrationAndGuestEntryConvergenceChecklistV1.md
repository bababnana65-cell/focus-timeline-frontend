# 登录/注册入口文案与游客入口收口清单 v1

本文用于冻结“游客先使用，账号后验证”的入口语义，供前端、后端和产品统一对照。

## 1. 核心产品语义

统一语义：

- 用户可以先以游客身份使用应用
- 只有在需要账号能力时，才触发手机号验证码验证
- 验证码通过后：
  - 若账号已存在，则直接登录
  - 若账号不存在，则自动创建账号并登录

对用户来说，这不是“两步：注册 + 登录”，而是一条统一的“手机号验证进入账号态”流程。

## 2. 总体结论

后续入口和文案按以下原则收口：

1. 新用户首次打开应用，不再强制进入“注册/登录页”
2. 默认直接进入推荐页，以游客身份开始使用
3. `RegistrationGateScreen` 或同类页面不再是启动必经页
4. 所有账号门禁改成“按需触发”
5. 页面文案不再强调“注册”，统一改为“手机号验证 / 登录以同步”语义

## 3. 游客默认可做的事

游客默认可用：

- 浏览推荐
- 打开时间轴
- 搜索专题/时间线节点
- 分享查看
- 本地关注专题（最多 5 个）
- 本地创建专题的过渡体验（后续按阶段 7 继续替换）

游客默认不可获得的正式账号能力：

- 跨设备同步
- 服务端长期保存的关注关系
- 更高 follow quota
- 付费能力

## 4. 账号验证触发时机

建议只在以下场景触发手机号验证：

1. 游客关注第 6 个专题
2. 游客创建第 6 个专题
3. 用户主动点头像 / 账号入口
4. 用户主动点“登录以同步”
5. 后续确有必要的账号能力场景

不建议再采用：

- 启动即强制登录
- 使用前必须注册
- 先注册才能浏览推荐/时间轴

## 5. 启动流收口

正式规则：

- App 启动默认进入 `HomeShell`
- 默认首页为“推荐”
- 游客状态下允许直接浏览

不再采用：

- 未登录 -> 必须先到 `RegistrationGateScreen`
- “注册/登录”作为启动强门禁

## 6. 页面与入口文案建议

### 6.1 不建议继续使用的文案

- 注册
- 注册新账号
- 登录 / 注册
- 先注册才能进入
- 注册后可使用

### 6.2 建议替换成的文案

主入口/按钮：

- 手机号验证
- 登录以同步
- 继续
- 立即进入
- 验证并继续

辅助说明：

- 首次使用将自动创建账号
- 登录后可同步关注和专题
- 登录后可提升关注上限
- 当前以游客身份使用

### 6.3 登录弹层标题建议

- 登录以继续
- 验证手机号
- 登录后同步你的关注和专题

不建议标题：

- 注册
- 注册账号
- 登录 / 注册

## 7. 前端收口清单

前端后续应按下面清单逐项收口：

1. 启动默认进入推荐页
2. `RegistrationGateScreen` 改成按需触发，不再是启动必经页
3. 游客态页面可以展示：
   - 当前以游客身份使用
   - 登录后可同步关注和专题
4. 第 6 个关注 / 第 6 个新建专题时，统一触发手机号验证引导
5. 登录成功后：
   - 自动 merge guest follows
   - 刷新“我的关注”
   - 清理本地 guest 关系缓存
6. 所有“注册”文案逐步替换成“手机号验证 / 登录以同步”

## 8. 后端配合前提

后端当前应继续保持以下语义稳定：

1. 验证码登录 = 登录或自动创建账号
2. `GET /users/capabilities` 可匿名返回 guest 能力
3. `POST /users/merge-guest-follows` 在登录后可用
4. 推荐、时间轴、分享查看等接口允许匿名读取

## 9. 推荐的游客提示语

### 游客首页轻提示

- 当前以游客身份使用
- 登录后可同步关注和专题

### 游客关注上限提示

- 游客最多可关注 5 个专题，登录后可关注 10 个

### 免费账号上限提示

- 当前最多可关注 10 个专题，升级后可关注 50 个

### 登录成功提示

- 已同步你的关注
- 首次使用已自动创建账号

## 10. 执行顺序建议

推荐执行顺序：

1. 先冻结产品语义
2. 再统一入口与文案
3. 再统一触发时机
4. 最后做轻量 UI 提示优化

## 11. 当前不做的事

当前阶段不在此文档内扩展：

- 支付订阅流程
- 多身份登录扩展文案
- 阶段 7 的 guest create 完整 UI
- 大规模视觉改版

## 12. 版本约束

当前文档为 `v1` 基线。

若后续产品规则变化：

- 新增 `v2`
- 不直接覆盖 `v1`
