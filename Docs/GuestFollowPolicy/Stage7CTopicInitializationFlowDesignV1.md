# 阶段 7C：服务端专题初始化链路设计草案 v1

本文用于定义“服务端已创建专题”如何从空壳状态推进到可阅读状态的最小设计基线。

目标：

- 新建专题不能长期停留在 `draft / pending + 空 timeline`
- 服务端创建成功后，要进入初始化链路
- 后端把专题从“等待初始化”推进到“可阅读”
- 为后续重试、失败恢复和状态展示预留最小扩展位

---

## 一、阶段 7C 的核心目标

阶段 7A / 7B 已经解决了：

1. 登录用户创建专题走服务端
2. guest create 最小入口走服务端
3. 创建成功后拿到正式 `topicId`

但当前仍存在一个明显空缺：

- 专题创建出来后，很多时候仍然只有：
  - `status = draft`
  - `initializationState = pending`
  - `timeline.entries = []`

阶段 7C 的目标就是补上这条链路：

1. 创建成功
2. 服务端挂起初始化任务
3. 初始内容生成
4. 专题进入可读状态

---

## 二、最小状态语义

当前建议先统一这组最小状态，不要求一次性暴露复杂任务平台。

### 1. `draft / pending`

含义：

- 专题基础记录已创建
- 初始化尚未开始或刚进入队列
- timeline 可能为空

前端建议展示：

- `正在准备中`

### 2. `draft / running`

含义：

- 初始化任务已开始执行
- 正在做定义扩写、检索、初始内容整理

前端建议展示：

- `正在生成初始内容`

### 3. `active / ready`

含义：

- 初始化已完成
- 已具备基础可阅读 timeline

前端建议展示：

- 正常阅读态

### 4. `draft / failed`

含义：

- 初始化任务失败
- 专题仍存在，但当前没有可用初始内容

前端建议展示：

- `初始化失败，请稍后重试`

---

## 三、初始化任务的最小语义

创建成功后，后端后续至少应处理这些事情：

1. 主题定义扩写
2. 初始检索
3. 初始 timeline 生成
4. 初始 sources / summary 整理

第一版不要求这几步全都非常智能，但至少要有一条明确的异步初始化链路。

---

## 四、最小后端流程

建议最小流程如下：

1. `POST /topics/create` 成功
2. 写入基础 `Topic` 记录
3. 返回：
   - `topic`
   - `followed`
   - `capabilities`
   - `initializationState = pending`
4. 后端异步触发初始化任务
5. 初始化任务执行：
   - 读取 `TopicDefinition`
   - 扩写定义或整理搜索 hint
   - 生成初始 timeline entries
   - 写回 sources / summary / counters
6. 成功后把状态推进为：
   - `status = active`
   - `initializationState = ready`
7. 失败则写为：
   - `status = draft`
   - `initializationState = failed`

---

## 五、最小返回语义

前端后续会依赖至少这几个字段：

### `topic.status`

- `draft`
- `active`

### `topic.initializationState`

- `pending`
- `running`
- `ready`
- `failed`

### timeline 层

- 如果 `initializationState != ready`
- `entries` 可以为空
- 前端不应把“空 timeline”直接当成永久无内容

---

## 六、失败与重试语义

第一版建议先支持最小失败可见性，不强求完整任务调度系统。

### 失败时

至少需要做到：

- 专题仍然存在
- 前端可看到 `failed`
- 不静默卡在 `pending`

### 重试

第一版可以先不开放前端主动重试按钮，
但后端设计上应允许：

- 手工重跑
- 后续补自动重试

---

## 七、当前阶段边界

这份草案当前只定义：

- 状态语义
- 初始化链路目标
- 前后端的最小职责边界

当前不要求：

- 完整任务平台
- 完整 AI workflow
- 推送系统
- 后台任务监控平台

---

## 八、前端最终依赖的页面语义

后续前端页面最终会依赖这三种状态：

1. `正在准备中`
2. `初始化失败`
3. `可正常阅读`

也就是说：

- “空 timeline”只是当前某个初始化阶段的表现
- 不是最终产品语义

---

## 九、推荐落地顺序

建议按这个顺序推进：

1. 先冻结状态语义
2. 后端补最小初始化任务骨架
3. 后端补状态推进
4. 前端接 `running / failed / ready`
5. 最后再考虑重试和更复杂调度

---

## 十、一句话版本

**阶段 7C 的核心，就是让服务端已创建的专题从 `draft / pending` 空壳状态推进到 `active / ready` 可阅读状态，并把“正在准备中 / 初始化失败 / 可阅读”的最小状态链路跑通。**
